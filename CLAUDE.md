# もちぽよバトラーズ 開発ガイド（AIアシスタント向け）

かわいいぽっちゃり系モンスター「もちぽよ」を収集・育成・バトルするブラウザゲーム。
このファイルは、開発を引き継ぐAIアシスタントが最初に読む前提で書かれている。

## 体制（3者分担）

- **ユーザー（hrmts）**: プロデューサー。方針決定、画像の透過調整などの手作業、投げ銭・Vercelアカウント管理
- **チャッピー（ChatGPT）**: 仕様書とイラストの供給元。ユーザー経由でMDファイルやzipが届く
- **あなた（Claude Code）**: 実装のオーケストレーター。詳細は下記「作業の進め方」

## 技術構成

- HTML / CSS / Vanilla JavaScript（ES Modules）。ビルドなし、フレームワークなし、**Canvasは使わない**
- セーブ: LocalStorage `mochipoyo_battlers_save_v2`（バージョン付き。互換はnormalizeSaveDataのベースマージで維持し、安易にキーを上げてセーブを消さない）
- デプロイ: GitHub `hrmtsyshkz-debug/mochipoyo-battlers` に push → Vercel自動デプロイ → https://mochipoyo-battlers.vercel.app
- 効果音: `js/audio.js` でWeb Audio合成（音源ファイルなし）。QR生成: `js/vendor/qrcode-generator.js`（同梱、CDN禁止）

## アーキテクチャの要点

- `data/` にゲームデータを分離（species 30種 / skills / items / stages）。**データ駆動を崩さない**
- モンスターはポケモン方式: 1進化形態 = 1図鑑枠 = 1 species。インスタンスは `speciesId` を持ち、進化で `speciesId` が変わる
- 画像は `assets/monsters/icon/`（丸枠用256px）と `full/`（512px）の2系統。参照チェーン: icon表示 = icon→full→emoji、full表示 = full→icon→emoji。**画像が無くても絵文字フォールバックで絶対に壊れないこと**
- ファイル名規則: `{dexNo}_{slug}.png`（例: `011_mochimochina.png`）
- 主要システム: ボス進化（レベル+専用アイテム）、満福度（ボス周回、`js/manpuku.js`）、バトルミニゲーム（`js/minigames.js`）、QRフレンド対戦（`js/challenge.js`）

## 作業の進め方（ユーザーとの合意事項）

1. **実装はサブエージェント（Sonnet）に委譲し、メインのあなたは仕様整理と品質検証に徹する**（推論コスト最適化のため）。ただし数行の修正はインラインでやる方が安い
2. サブエージェントには「再委譲禁止・自分で実装せよ」と明記する（過去に再委譲して空振りした事故あり）
3. **検証は必ず自分でブラウザ実測**: preview（launch.jsonの `mochipoyo-static-server`、ポート5174）で一連のフローを操作し、境界値（29/30など）・改ざんペイロード・旧セーブ互換・コンソールエラーゼロを確認してからコミット
4. コミット・プッシュはレビュー後にあなたが行う（push=本番反映なので検証前に押さない）。デプロイ後はcurlで本番配信を確認
5. チャッピーの仕様書と実装の齟齬（番号ズレ・用語の新旧混在など）は頻発する。**名前を正として突き合わせ、裁定をユーザーに明示してから進める**

## 画像パイプライン（あなたの担当作業）

1. 納品zipは展開して内容・解像度・透過（四隅アルファ）・既存とのハッシュ一致を確認
2. **オリジナルは `G:\Claude\mochipoyo-art-originals\` に保管**（リポジトリ外。新番号命名で。iconは `_icon` サフィックス）
3. ゲーム用に PowerShell + System.Drawing で加工: full 512px / icon 256px（正方形納品ならリサイズのみ。縦長ならキャラ中心クロップ）
4. `data/monsters.js` の images パスを設定（未納品は null のまま）
5. ユーザーが画像を直接編集することがある（透過調整など）。「画像いじった」と言われたらオリジナルフォルダの更新日時で差分を特定して再処理
